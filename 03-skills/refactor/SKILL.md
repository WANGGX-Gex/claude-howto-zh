# 代码重构技能

基于 Martin Fowler 的《重构：改善既有代码的设计》（第 2 版）的系统化重构方法。本技能强调由测试支撑的安全、增量式变更。

> "重构是在不改变代码外部行为的前提下，改善其内部结构的过程。" —— Martin Fowler

## 核心原则

1. **行为保持**：外部行为必须保持不变
2. **小步骤**：做微小的、可测试的变更
3. **测试驱动**：测试是安全网
4. **持续进行**：重构是持续的，而非一次性事件
5. **协作**：每个阶段需要用户批准

## 工作流概览

```
阶段 1：研究与分析
    ↓
阶段 2：测试覆盖率评估
    ↓
阶段 3：代码异味识别
    ↓
阶段 4：重构计划创建
    ↓
阶段 5：增量实施
    ↓
阶段 6：审查与迭代
```

---

## 阶段 1：研究与分析

### 目标
- 理解代码库结构和目的
- 确定重构范围
- 收集业务需求的上下文

### 向用户提出的问题
开始前，澄清：

1. **范围**：哪些文件/模块/函数需要重构？
2. **目标**：你试图解决什么问题？（可读性、性能、可维护性）
3. **约束**：有哪些区域不应更改？
4. **时间压力**：这是否阻塞了其他工作？
5. **测试状态**：测试是否存在？是否通过？

### 操作
- [ ] 阅读并理解目标代码
- [ ] 识别依赖和集成
- [ ] 记录当前架构
- [ ] 记录任何现有的技术债务标记（TODO、FIXME）

### 输出
向用户展示发现：
- 代码结构摘要
- 识别的问题区域
- 初步建议
- **请求批准继续**

## 阶段 2：测试覆盖率评估

### 为什么测试很重要
> "没有测试的重构就像不系安全带开车。" —— Martin Fowler

测试是安全重构的**关键使能条件**。没有测试，你就有引入缺陷的风险。

### 评估步骤

1. **检查现有测试**
   ```bash
   # Look for test files
   find . -name "*test*" -o -name "*spec*" | head -20
   ```

2. **运行现有测试**
   ```bash
   # JavaScript/TypeScript
   npm test

   # Python
   pytest -v

   # Java
   mvn test
   ```

3. **检查覆盖率（如果可用）**
   ```bash
   # JavaScript
   npm run test:coverage

   # Python
   pytest --cov=.
   ```

### 决策点：询问用户

**如果测试存在且通过：**
- 进入阶段 3

**如果测试缺失或不完整：**
提供选项：
1. 先编写测试（推荐）
2. 在重构过程中逐步添加测试
3. 不写测试直接进行（风险较高 - 需要用户确认）

**如果测试失败：**
- 停止。在重构前修复失败的测试
- 询问用户：是否应该先修复测试？

### 测试编写准则（如需要）

对于每个被重构的函数，确保测试覆盖：
- 正常路径（正常操作）
- 边界情况（空输入、null、边界值）
- 错误场景（无效输入、异常）

使用"红-绿-重构"循环：
1. 编写失败的测试（红）
2. 使其通过（绿）
3. 重构

## 阶段 3：代码异味识别

### 什么是代码异味？
代码中更深层问题的症状。它们不是缺陷，但表明代码可以改进。

### 需要检查的常见代码异味

完整目录请参见 [references/code-smells.md](references/code-smells.md)。

#### 快速参考

| 异味 | 征兆 | 影响 |
|-------|-------|--------|
| **过长方法** | 方法 > 30-50 行 | 难以理解、测试、维护 |
| **重复代码** | 相同逻辑出现在多处 | 缺陷修复需要在多处进行 |
| **过大的类** | 类有太多职责 | 违反单一职责原则 |
| **特性嫉妒** | 方法更多使用另一个类的数据 | 封装性差 |
| **基本类型偏执** | 过度使用基本类型而非对象 | 缺少领域概念 |
| **过长参数列表** | 方法有 4+ 个参数 | 难以正确调用 |
| **数据泥团** | 相同的数据项反复一起出现 | 缺少抽象 |
| **Switch 语句** | 复杂的 switch/if-else 链 | 难以扩展 |
| **过度预测** | "以防万一"的代码 | 不必要的复杂性 |
| **死代码** | 未使用的代码 | 混乱，维护负担 |

### 分析步骤

1. **自动分析**（如果脚本可用）
   ```bash
   python scripts/detect-smells.py <file>
   ```

2. **手动审查**
   - 系统地遍历代码
   - 记录每个异味的位置和严重程度
   - 按影响分类（严重/高/中/低）

3. **优先级排序**
   聚焦于以下异味：
   - 阻塞当前开发
   - 导致缺陷或混乱
   - 影响最常修改的代码路径

### 输出：异味报告

向用户展示：
- 已识别异味及位置列表
- 每个异味的严重程度评估
- 建议的优先顺序
- **请求批准优先级**

## 阶段 4：重构计划创建

### 选择重构技术

为每个异味选择适当的重构技术。

完整列表请参见 [references/refactoring-catalog.md](references/refactoring-catalog.md)。

#### 异味到重构技术的映射

| 代码异味 | 推荐的重构技术 |
|------------|---------------------------|
| 过长方法 | 提取方法、以查询替代临时变量 |
| 重复代码 | 提取方法、上移方法、形成模板方法 |
| 过大的类 | 提取类、提取子类 |
| 特性嫉妒 | 移动方法、移动字段 |
| 基本类型偏执 | 以对象替代基本类型、以类替代类型码 |
| 过长参数列表 | 引入参数对象、保持对象完整 |
| 数据泥团 | 提取类、引入参数对象 |
| Switch 语句 | 以多态替代条件表达式 |
| 过度预测 | 折叠继承体系、内联类、移除死代码 |
| 死代码 | 移除死代码 |

### 计划结构

使用 [templates/refactoring-plan.md](templates/refactoring-plan.md) 中的模板。

对每个重构：
1. **目标**：哪些代码将改变
2. **异味**：解决什么问题
3. **重构技术**：应用哪种技术
4. **步骤**：详细的微步骤
5. **风险**：可能出什么问题
6. **回滚**：如何撤销

### 分阶段方法

**关键**：逐阶段渐进式引入重构。

**阶段 A：快速收益**（低风险、高价值）
- 重命名变量以提高清晰度
- 提取明显的重复代码
- 移除死代码

**阶段 B：结构改进**（中等风险）
- 从长函数中提取方法
- 引入参数对象
- 将方法移至适当的类

**阶段 C：架构变更**（较高风险）
- 以多态替代条件表达式
- 提取类
- 引入设计模式

### 决策点：向用户展示计划

实施前：
- 展示完整的重构计划
- 解释每个阶段及其风险
- 获取每个阶段的明确批准
- **询问**："是否应该开始阶段 A？"

## 阶段 5：增量实施

### 黄金法则
> "变更 → 测试 → 绿灯？→ 提交 → 下一步"

### 实施节奏

对每个重构步骤：

1. **预检查**
   - 测试通过（绿灯）
   - 代码编译通过

2. **做一个小变更**
   - 遵循目录中的技术步骤
   - 保持变更最小

3. **验证**
   - 立即运行测试
   - 检查编译错误

4. **如果测试通过（绿灯）**
   - 使用描述性消息提交
   - 进入下一步

5. **如果测试失败（红灯）**
   - 立即停止
   - 撤销变更
   - 分析出了什么问题
   - 如不确定则询问用户

### 提交策略

每个提交应该是：
- **原子性的**：一个逻辑变更
- **可回退的**：容易回滚
- **描述性的**：清晰的提交消息

提交消息示例：
```
refactor: Extract calculateTotal() from processOrder()
refactor: Rename 'x' to 'customerCount' for clarity
refactor: Remove unused validateOldFormat() method
```

### 进度报告

每个子阶段后，向用户报告：
- 所做的变更
- 测试是否仍然通过？
- 遇到的任何问题
- **询问**："继续下一批？"

## 阶段 6：审查与迭代

### 重构后清单

- [ ] 所有测试通过
- [ ] 无新警告/错误
- [ ] 代码编译成功
- [ ] 行为未改变（手动验证）
- [ ] 文档已更新（如需要）
- [ ] 提交历史清晰

### 指标对比

重构前后运行复杂度分析：
```bash
python scripts/analyze-complexity.py <file>
```

展示改进：
- 代码行数变化
- 圈复杂度变化
- 可维护性指数变化

### 用户审查

展示最终结果：
- 所有变更的摘要
- 前后代码对比
- 指标改进
- 剩余技术债务
- **询问**："你对这些变更满意吗？"

### 后续步骤

与用户讨论：
- 是否还有更多异味需要处理？
- 是否安排后续重构？
- 是否将类似变更应用到其他地方？

## 重要准则

### 何时停止并询问

在以下情况下始终暂停并咨询用户：
- 不确定业务逻辑
- 变更可能影响外部 API
- 测试覆盖率不足
- 需要重大架构决策
- 风险级别增加
- 遇到意外的复杂性

### 安全规则

1. **绝不在没有测试的情况下重构**（除非用户明确确认风险）
2. **绝不做大变更** - 分解为微小步骤
3. **绝不跳过测试运行** - 每次变更后都要测试
4. **测试失败时绝不继续** - 先修复或回滚
5. **绝不假设** - 有疑问时就问

### 不应该做的事

- 不要将重构与功能添加混合
- 不要在生产紧急情况下重构
- 不要重构你不理解的代码
- 不要过度工程化 - 保持简单
- 不要一次重构所有东西

## 快速入门示例

### 场景：带有重复代码的过长方法

**重构前：**
```javascript
function processOrder(order) {
  // 150 lines of code with:
  // - Duplicated validation logic
  // - Inline calculations
  // - Mixed responsibilities
}
```

**重构步骤：**

1. **确保** processOrder() 有测试
2. **提取** 验证逻辑到 validateOrder()
3. **测试** - 应该通过
4. **提取** 计算逻辑到 calculateOrderTotal()
5. **测试** - 应该通过
6. **提取** 通知逻辑到 notifyCustomer()
7. **测试** - 应该通过
8. **审查** - processOrder() 现在编排 3 个清晰的函数

**重构后：**
```javascript
function processOrder(order) {
  validateOrder(order);
  const total = calculateOrderTotal(order);
  notifyCustomer(order, total);
  return { order, total };
}
```
