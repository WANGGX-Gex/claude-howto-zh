# 检查点示例

在 Claude Code 中高效使用检查点的真实案例。

注意：检查点随每次用户提示词自动创建。你无需手动保存。要回退，请连按两次 `Esc`（`Esc+Esc`）或使用 `/rewind` 打开检查点浏览器。

## 示例 1：数据库迁移

### 场景
你需要从 MySQL 迁移到 PostgreSQL，但想同时探索直接迁移和渐进式双写两种方案。

### 工作流

```
用户：让我们实现从 MySQL 到 PostgreSQL 的直接迁移
      [自动创建检查点 A]

Claude：我将实现直接迁移...
[更改数据库驱动、更新查询、修改模式]

用户：运行测试
      [自动创建检查点 B]

Claude：正在运行测试...
❌ 15 个测试失败 - 连接池问题

用户：这个方案有问题。让我们尝试双写模式。
[用户按 Esc+Esc，选择检查点 A，选择"恢复代码和对话"]

用户：现在实现双写模式，同时写入两个数据库

Claude：我将实现双写迁移...
[实现双写层、渐进式迁移策略]

用户：运行测试

Claude：正在运行测试...
✅ 所有测试通过

用户：双写方案明显更安全。让我们提交这个。
```

## 示例 8：使用从检查点总结

### 场景
经过长时间的调试会话后，你想压缩对话同时保留上下文。

### 工作流

```
用户：[经过 20+ 条消息的调试和探索后]

[用户按 Esc+Esc，选择一个早期检查点，选择"从此处总结"]
[可选提供指令："聚焦于我们尝试了什么以及什么有效"]

Claude：[从该时间点开始生成对话摘要]
[原始消息保留在转录记录中]
[摘要替换可见的对话，减少上下文窗口使用]

用户：现在让我们继续使用有效的方案。
```

## 核心要点

1. **检查点是自动的**：每次用户提示词都会创建检查点 -- 无需手动保存
2. **使用 Esc+Esc 或 /rewind**：这是访问检查点浏览器的两种方式
3. **选择正确的恢复选项**：根据需要选择恢复代码、对话、两者或总结
4. **不要害怕实验**：检查点让你可以安全地尝试激进的更改
5. **结合 git 使用**：用检查点进行探索，用 git 保存确定的工作
6. **总结长会话**：使用"从此处总结"来保持对话的可管理性
