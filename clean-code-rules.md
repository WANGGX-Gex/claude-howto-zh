# AI 代码生成的整洁代码规则

这些规则指导代码生成，以产出可维护的、专业品质的代码。

## 有意义的命名
- 使用揭示意图的名称，解释某事物存在的原因
- 避免误导和无意义的区分（如 `data`、`info`、`manager`）
- 使用可读的、可搜索的名称
- 类名：名词（如 `UserAccount`、`PaymentProcessor`）
- 方法名：动词（如 `calculateTotal`、`sendEmail`）
- 避免心理映射和编码（匈牙利命名法、前缀）

## 函数
- 保持函数简短（理想情况下少于 20 行）
- 只做一件事——单一职责原则
- 每个函数一个抽象层级
- 限制参数：0-2 个为理想，最多 3 个，避免布尔标志参数
- 无副作用——函数应该做其名称所表示的事
- 分离命令（改变状态）和查询（返回信息）
- 优先使用异常而非错误码

## 注释
- 代码应该自解释——尽可能避免注释
- 好的注释：法律信息、警告、TODO、公共 API 文档
- 坏的注释：冗余的、误导的、或为糟糕代码辩解的
- 永远不要注释掉代码——删除它（版本控制会保留历史）
- 如果需要注释，考虑重构代码

## 格式
- 保持文件小而专注
- 垂直格式化：相关概念靠近，空行分隔概念
- 水平格式化：限制行长度（80-120 字符）
- 使用一致的缩进和团队风格
- 将相关函数分组在一起

## 对象和数据结构
- 对象：在抽象背后隐藏数据，通过方法暴露行为
- 数据结构：暴露数据，拥有最少的行为
- 迪米特法则：只与直接朋友交流，避免 `a.getB().getC().doSomething()`
- 不要盲目地通过 getter/setter 暴露内部结构

## 错误处理
- 使用异常，而非返回码或错误标志
- 当代码可能失败时，先写 `try-catch-finally`
- 在异常消息中提供上下文
- 不要返回 `null`——返回空集合或使用 Optional/Maybe
- 不要传递 `null` 作为参数

## 类
- 小型类：以职责而非行数衡量
- 单一职责原则：只有一个变更的理由
- 高内聚：类变量被多个方法使用
- 低耦合：类之间依赖最少
- 开闭原则：对扩展开放，对修改关闭

## 单元测试
- 快速、独立、可重复、自验证、及时（F.I.R.S.T.）
- 每个测试一个断言（或一个概念）
- 测试代码质量等同于生产代码质量
- 可读的测试名称，描述被测试的内容
- Arrange-Act-Assert 模式

## 代码质量原则
- **DRY（不要重复自己）**: 无重复
- **YAGNI（你不会需要它）**: 不要为假想的未来构建
- **KISS（保持简单）**: 避免不必要的复杂性
- **童子军规则**: 让代码比你发现时更干净

## 要避免的代码坏味道
- 过长的函数或类
- 重复代码
- 死代码（未使用的变量、函数、参数）
- 特性嫉妒（方法对其他类更感兴趣）
- 不恰当的亲密关系（类彼此了解太多）
- 过长的参数列表
- 基本类型偏执（过度使用基本类型而非小对象）
- Switch/case 语句（考虑多态）
- 临时字段（类变量只在某些时候使用）

## 并发
- 将并发代码与其他代码分离
- 限制同步/锁定数据的作用域
- 使用线程安全的集合
- 保持同步区域小
- 了解你的执行模型和原语

## 系统设计
- 将构建与使用分离（依赖注入）
- 使用工厂、构建器创建复杂对象
- 面向接口编程，而非面向实现
- 优先组合而非继承
- 当设计模式能简化时才应用，而非为了炫技

## 重构
- 持续重构，而非大批量重构
- 重构前后始终保持测试通过
- 小步骤：一次一个变更
- 常见重构：提取方法、重命名、移动、内联

## 文档
- 自文档化代码 > 注释 > 外部文档
- 公共 API 需要清晰的文档
- 在文档中包含示例
- 让文档靠近代码（理想情况下在代码中）

---

**核心理念**: 代码被阅读的次数是被编写次数的 10 倍。为可读性和可维护性优化，而非为了炫技。
