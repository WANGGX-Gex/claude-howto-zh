# 安全政策

## 概述

Claude How To 项目的安全性对我们非常重要。本文档概述了我们的安全实践，并描述了如何负责任地报告安全漏洞。

## 受支持的版本

我们为以下版本提供安全更新：

| 版本 | 状态 | 支持截止 |
|------|------|---------|
| 最新版（main） | 积极维护 | 当前 + 6 个月 |
| 1.x 版本 | 积极维护 | 直到下一个主版本 |

**注意**: 作为一个教育性指南项目，我们专注于维护当前的最佳实践和文档安全性，而非传统的版本支持。更新直接应用到 main 分支。

## 安全实践

### 代码安全

1. **依赖管理**
   - 所有 Python 依赖都在 `requirements.txt` 中固定版本
   - 通过 dependabot 和手动审查进行定期更新
   - 每次提交都使用 Bandit 进行安全扫描
   - 预提交钩子（Hook）用于安全检查

2. **代码质量**
   - 使用 Ruff 进行代码检查，捕获常见问题
   - 使用 mypy 进行类型检查，防止类型相关漏洞
   - 预提交钩子强制执行标准
   - 所有更改在合并前经过审查

3. **访问控制**
   - `main` 分支启用分支保护
   - 合并前需要审查通过
   - 合并前状态检查必须通过
   - 仓库写入权限受限

### 文档安全

1. **示例中无密钥**
   - 所有示例中的 API 密钥都是占位符
   - 凭据从不硬编码
   - `.env.example` 文件展示所需变量
   - 关于密钥管理的明确警告

2. **安全最佳实践**
   - 示例展示安全模式
   - 文档中突出显示安全警告
   - 链接到官方安全指南
   - 在相关章节讨论凭据处理

3. **内容审查**
   - 所有文档经过安全问题审查
   - 贡献指南中包含安全注意事项
   - 验证外部链接和引用

### 依赖安全

1. **扫描**
   - Bandit 扫描所有 Python 代码的漏洞
   - 通过 GitHub 安全警报进行依赖漏洞检查
   - 定期手动安全审计

2. **更新**
   - 安全补丁及时应用
   - 主版本谨慎评估
   - 变更日志包含安全相关更新

3. **透明度**
   - 安全更新在提交中记录
   - 漏洞披露负责任地处理
   - 适当时发布公开安全公告

## 报告漏洞

### 我们关注的安全问题

我们重视以下方面的报告：
- 脚本或示例中的**代码漏洞**
- Python 包中的**依赖漏洞**
- 代码示例中的**加密问题**
- 文档中的**认证/授权缺陷**
- 配置示例中的**数据暴露风险**
- **注入漏洞**（SQL、命令等）
- **SSRF/XXE/路径遍历**问题

### 超出范围的安全问题

以下不在本项目范围内：
- Claude Code 本身的漏洞（请向 Anthropic 报告）
- 外部服务或库的问题（请向上游报告）
- 社会工程或用户教育（不适用于本指南）
- 没有概念验证的理论漏洞
- 已通过官方渠道报告的依赖漏洞

## 如何报告

### 私密报告（推荐）

**对于敏感安全问题，请使用 GitHub 的私密漏洞报告：**

1. 访问: https://github.com/luongnv89/claude-howto/security/advisories
2. 点击"Report a vulnerability"
3. 填写漏洞详情
4. 包含：
   - 漏洞的清晰描述
   - 受影响的组件（文件、章节、示例）
   - 潜在影响
   - 复现步骤（如适用）
   - 建议的修复方案（如果你有的话）

**后续流程：**
- 我们将在 48 小时内确认收到
- 我们将调查并评估严重性
- 我们将与你合作开发修复方案
- 我们将协调披露时间线
- 我们将在安全公告中为你署名（除非你希望匿名）

### 公开报告

对于非敏感问题或已经公开的问题：

1. **创建 GitHub Issue** 并添加 `security` 标签
2. 包含：
   - 标题: `[SECURITY]` 加上简要描述
   - 详细描述
   - 受影响的文件或章节
   - 潜在影响
   - 建议的修复方案

## 漏洞响应流程

### 评估（24 小时）

1. 我们确认收到报告
2. 我们使用 [CVSS v3.1](https://www.first.org/cvss/v3.1/specification-document) 评估严重性
3. 我们确定是否在范围内
4. 我们与你联系并提供初步评估

### 开发（1-7 天）

1. 我们开发修复方案
2. 我们审查和测试修复方案
3. 我们创建安全公告
4. 我们准备发布说明

### 披露（根据严重性而定）

**严重（CVSS 9.0-10.0）**
- 立即发布修复
- 发布公开公告
- 提前 24 小时通知报告者

**高（CVSS 7.0-8.9）**
- 在 48-72 小时内发布修复
- 提前 5 天通知报告者
- 发布时附带公开公告

**中等（CVSS 4.0-6.9）**
- 在下次常规更新中发布修复
- 发布时附带公开公告

**低（CVSS 0.1-3.9）**
- 在下次常规更新中包含修复
- 发布时附带公告

### 发布

我们发布的安全公告包括：
- 漏洞描述
- 受影响的组件
- 严重性评估（CVSS 分数）
- 修复版本
- 变通方案（如适用）
- 报告者署名（经同意）

## 报告者最佳实践

### 报告前

- **验证问题**: 你能持续复现吗？
- **搜索已有 Issue**: 是否已经被报告？
- **检查文档**: 是否有安全使用的指南？
- **测试修复**: 你建议的修复方案有效吗？

### 报告时

- **具体明确**: 提供确切的文件路径和行号
- **包含上下文**: 为什么这是安全问题？
- **展示影响**: 攻击者能做什么？
- **提供步骤**: 我们如何复现？
- **建议修复**: 你会如何修复？

### 报告后

- **保持耐心**: 我们的资源有限
- **及时回应**: 快速回答后续问题
- **保持保密**: 在修复前不要公开披露
- **尊重协调**: 遵循我们的披露时间线

## 安全头部和配置

### 仓库安全

- **分支保护**: main 分支的更改需要 2 个批准
- **状态检查**: 所有 CI/CD 检查必须通过
- **CODEOWNERS**: 关键文件的指定审查者
- **签名提交**: 推荐贡献者使用

### 开发安全

```bash
# 安装预提交钩子
pre-commit install

# 在本地运行安全扫描
bandit -c pyproject.toml -r scripts/
mypy scripts/ --ignore-missing-imports
ruff check scripts/
```

### 依赖安全

```bash
# 检查已知漏洞
pip install safety
safety check

# 或使用 pip-audit
pip install pip-audit
pip-audit
```

## 贡献者安全指南

### 编写示例时

1. **永远不要硬编码密钥**
   ```python
   # 不推荐
   api_key = "sk-1234567890"

   # 推荐
   api_key = os.getenv("API_KEY")
   ```

2. **警告安全影响**
   ```markdown
   **安全注意**: 永远不要将 `.env` 文件提交到 git。
   立即添加到 `.gitignore`。
   ```

3. **使用安全默认值**
   - 默认启用认证
   - 在适用处使用 HTTPS
   - 验证和清理输入
   - 使用参数化查询

4. **记录安全注意事项**
   - 解释为什么安全很重要
   - 展示安全与不安全的模式对比
   - 链接到权威来源
   - 突出显示警告

### 审查贡献时

1. **检查暴露的密钥**
   - 扫描常见模式（api_key=, password=）
   - 审查配置文件
   - 检查环境变量

2. **验证安全编码实践**
   - 没有硬编码的凭据
   - 适当的输入验证
   - 安全的认证/授权
   - 安全的文件处理

3. **测试安全影响**
   - 这能被滥用吗？
   - 最坏的情况是什么？
   - 有边界情况吗？
